Claude Codeに実装を頼んで「できました」と言われたとき、ちゃんと動くか試したくなります。動いた。では終わりにしていいか。——最初のうちはそれで済ませていたのですが、しばらくして「これは違うな」と気づきました。

動くことと正しいことは別の話で、AIが書いたコードには「動くが問題を抱えている」ものが普通に混じっています。以来、レビューするときに見るべき観点をある程度固めて使っています。

## 前提: AIは正直ではない

誤解のないように言うと、AIは嘘をつくというより楽観的です。「これで動きますか」と聞くと「動きます」と言いがちで、「こういうケースは考慮しましたか」と聞いてはじめて「あ、そこは漏れていました」となることがよくあります。

つまりこちらから問わないと出てこない情報がある。レビューとはその問いを立てる行為だと思っています。

## 1. 動いていることと意図通りであることを分けて確認する

最初に確認するのは、出力された結果が仕様と一致しているかです。テストケースを実行すれば動作確認はできますが、そのテスト自体をAIが書いていると、ハッピーパス（正常系）だけ通る状態になっていることがあります。

実際にあったのが、リスト処理を頼んだら空リストの扱いを考慮していなかったケース。テストは通る。ただし空リストが来ると `undefined` が返ってくる実装になっていました。テストコードも「リストに要素がある前提」でしか書かれていませんでした。

AIは「テストをパスするコード」を書くのは得意ですが、「テストが正しいかどうか」には無頓着なことがあります。

## 2. AIが省略したエッジケースを探す

AIに実装を頼むと、典型的なケースはきれいに処理します。問題は典型的でないケースです。

組み込みの仕事をしてきた経験からすると、バグのほとんどはエッジにあります。入力が0のとき、リストが空のとき、ネットワークが切れたとき、タイムアウトしたとき——こういうところを手を動かして確認するのが習慣になっています。

AIに「このコードで考慮できていないケースはあるか」と問いかけると、自分で気づいていなかった穴を挙げてくることがあります。ただしこれも全部は出てこないので、最後は自分で想像するしかない。

```
確認している観点の例:
- 入力が空、null、または想定外の型のとき
- 非同期処理でエラーが起きたとき
- 繰り返し呼び出されたとき（副作用の累積）
- データ量が大きいとき（ループのネストがないか）
```

## 3. 責務の境界を見る

AIはコードをひとつの関数に詰め込みがちです。「ユーザー情報を取得して整形してメールを送る」を一気に頼むと、その3つが1関数に入ってくることがよくあります。動くのですが、後から変えにくい。

特に気にするのが、外部への依存が関数の奥に埋まっているパターンです。データベースの呼び出しやHTTPリクエストがビジネスロジックの中に直接書かれていると、テストが難しくなりますし、接続先を変えるときに影響範囲が読めなくなります。

ここは動作確認では見えない問題なので、コードを読んで判断するしかありません。

## 4. セキュリティの穴を確認する

AIはリクエストに素直に応えようとするので、安全側に倒すより利便性を優先した実装になることがあります。

確認するようにしているのはこのあたりです。

**SQLを直接組み立てていないか**
文字列連結でSQLを作るコードが出てきたことがあります。パラメータバインドを使うよう修正しました。

**入力値の検証があるか**
外部から来る値をそのまま使っていないか。数値として扱う前にパースしているか。

**認証・認可の確認**
エンドポイントに認証チェックが入っているか。管理者用の処理がログインユーザーなら誰でも呼べる状態になっていないか。

これはAIに「このコードにセキュリティ上の問題はあるか」と聞くのも有効です。自分が見落としている視点を出してくることがあります。

## 5. 後から変えにくい部分を見抜く

これが一番難しいです。今動いているかどうかではなく、3ヶ月後に変更が入ったときに困るかどうかを見る作業です。

経験的に引っかかりを感じるのはこういうところです。

- ハードコードされた値（URLや設定値が直書きされている）
- 処理の流れが追いにくいネストの深さ
- 名前から何をするか読み取れない変数・関数名
- 似たコードが複数箇所に散らばっている

この観点でAIに相談するなら「このコードを6ヶ月後に別の人が変更するとしたら、どこが難しくなりそうか」という聞き方が割と当たりを引きます。

## レビューにかける時間の感覚

量が増えると全部を読むわけにもいかないので、場所によって見る深さを変えています。

| 箇所 | 優先度 | 主な確認内容 |
|------|--------|------------|
| 外部入力を受け取る部分 | 高 | 検証、エスケープ、認証 |
| データベース操作 | 高 | SQLインジェクション、トランザクション |
| 新しいファイル・関数 | 中 | 責務の適切さ、命名 |
| 既存コードの修正部分 | 中 | 影響範囲、意図との一致 |
| ユーティリティ的な処理 | 低 | エッジケース |

## AIに問いかけながらレビューする

一人でレビューするより、AIに問いかけながらやる方が効率的だと感じています。自分では思いつかない観点を出してくることがあるし、その場でコードの意図を確認できます。

ただし「このコードに問題はあるか」という聞き方だと、表面的な指摘しか返ってこないことがあります。「このコードを10万ユーザーが使ったとしたら」「悪意のある入力を受けたとしたら」のように状況を仮定した聞き方の方が、掘り下げた回答を引き出せます。

AIが書いたコードをAIと一緒にレビューするのは、少し奇妙な話に聞こえるかもしれません。ですが実際には有効で、書いたAIと別の文脈でやり取りする形になるので、死角を補いやすいと思っています。
