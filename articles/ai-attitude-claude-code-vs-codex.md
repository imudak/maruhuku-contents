Claude CodeにCodexをSkillとして組み込んで使い始めてから、気づいたことがあります。「Codexに相談しよう」と思ったことが一度もない。自然と「Codexはレビュー用」「相談はClaude Code」という使い分けになっていました。

性能の差で分けていたわけではありません。**態度の差**で分けていました。

## Skillで2つを並べてわかったこと

以前、Claude CodeにCodexをSkillとして設定した話を書きました。

https://zenn.dev/imudak/articles/claude-code-codex-skill

セッション中に `/codex` コマンドでCodex CLIを呼び出せるようにして、Claude Codeが書いたコードをCodexへ渡してレビューさせる——という使い方です。同じセッション内で2つのAIを並べて使うと、返答のスタイルの違いが見えてきました。

## 「口数の多さ」の違い

Claude Codeに何かを頼むと、前置きがあります。「このファイルを見ると〜という構造になっているので」「ご要望の通り〜しました。ただし注意点として〜」。聞いていないことも教えてくれます。

Codexは違います。「このコードの問題点は3つあります」で始まって、箇条書きで終わります。前置きがない。経緯も説明しない。

どちらが優れているという話ではなく、**想定しているインタラクションが違う**と感じました。

Claude Codeは対話が続くことを前提にしています。会話が積み上がるほど、コンテキストが深まるほど、精度が上がる設計です。

Codexは1往復を想定しています。コードを渡す、分析が返ってくる、以上。

## 「きつさ」がレビューとして機能する

最初は口数の少なさを「不親切」と感じていました。でも使っていると、コードレビューの場面では「きつさ」が機能すると気づきました。

「このコードはどう思う？」に対して、前置きなしに「ここがまずいです」と返ってくる方が、レビューとしては受け取りやすい。優しすぎると、何が問題かの輪郭がぼやけます。

逆に「この機能をどう実装するか一緒に考えてほしい」という場面では、コンテキストを蓄積しながら会話できるClaude Codeの方が向いています。

気づいたら、自分の中で使い分けが固まっていました。

| 用途 | ツール | 理由 |
|------|--------|------|
| 実装・設計・相談 | Claude Code | 会話として進める |
| コードレビュー・分析 | Codex | 答えを取り出す |

## 性能差ではなく「態度差」

この使い分けを「Claude Codeの方が賢い」「Codexの方が手軽」という軸で説明しようとすると、うまくいきません。

Claude Codeにレビューさせることも、Codexに設計相談することも、形の上ではできます。でも体験が違います。Claude Codeは「一緒に考えてくれる相手」で、Codexは「コードを見てもらう相手」という感覚が定着しました。

使い分けの基準が「性能」でも「機能」でもなく「態度」だというのは、AI普及期として面白い現象だと思います。

人間同士でも、相談相手と査読者を分けることはあります。同じくらいの能力を持っていても、やりとりのスタイルで適材適所が決まります。AIにも同じことが起き始めている気がします。

## 「どっちを使う？」への答え方が変わる

個人で使う分には、使っているうちに自然と使い分けができてきます。でもチーム全体への導入を考え始めると、「どっちを使えばいいか」という疑問が出てきます。

その答えを「こっちの方が賢いから」で出そうとすると難しい。「相談しやすさ」「レビューのきつさ」という軸の方が、実際の使い分けに即しています。

AI選定の基準が「何ができるか」から「どんな相手か」に変わりつつある——という感覚があります。

## まとめ

- Claude CodeとCodexの役割分担は、性能差でなく「態度（口数・返答スタイル）の違い」から生まれた
- コードレビューには「きつさ」が機能する
- AI選定の基準が「機能」から「インタラクションのスタイル」に移りつつある
- チーム導入では、この「態度差」が選定基準として重要になるかもしれない
