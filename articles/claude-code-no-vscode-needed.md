「Cursorで動かすのは情弱」——Xでそんな投稿を見かけました。煽り気味の物言いですが、言いたいことはわかります。Claude CodeはCLIツールなので、VS CodeやCursorを間に挟む意味がないのでは、ということです。

見かけた時、自分はそもそもVS Codeを開いていなかったので「まあそうだよな」という感想でした。ターミナルだけで開発を回している現状と、なぜそうなったかを書きます。

## 現在の開発環境

```text
WSL2（Ubuntu）
└── tmux
    ├── ウィンドウ0: claude（メインセッション）
    ├── ウィンドウ1: claude（サブエージェント用）
    ├── ウィンドウ2: claude（並列実行用）
    └── ウィンドウ3: ログ確認・監視
```

VS Codeは起動していません。外側にWindows Terminalがあるだけで、作業はすべてtmux内で完結しています。

もともと組み込み開発メインで、Visual Studioを長く使ってきました。Web系に手を出すようになってVS Codeに移り、それが当たり前になっていました。Claude Codeを本格的に使い始めてから、気づいたらVS Codeを開かなくなっていました。

## Claude Codeは何をやっているか

Claude CodeはIDEのような機能を、ツールとして持っています。

| 操作 | VS Code | Claude Code |
|------|---------|-------------|
| ファイル読み書き | エディタ | Read / Write ツール |
| コード検索 | Cmd+Shift+F | Grep / Glob ツール |
| Git操作 | ソース管理パネル | Bash ツール |
| コード編集・リファクタリング | エディタ + 補完 | Edit ツール（直接変更） |
| 構文ハイライト | ○ | ✕ |
| デバッガ | ○ | △（ログ確認のみ） |

構文ハイライトとデバッガはClaude Codeにありません。ただ、Claude Codeで開発していると、コードを目で追う機会が大幅に減ります。指示して、結果を確認して、次の指示を出す、の繰り返しです。デバッガが必要になる場面——スタック追跡が必要なクラッシュなど——は確かにあります。そういうときは素直にVS Codeを開きます。それ以外はターミナルの中で完結しています。

## ターミナルの利点：並列実行との相性

ターミナルを選んでいる理由のひとつが、**tmuxとの組み合わせ**です。

tmuxのウィンドウやペインに複数のClaude Codeセッションを並べて、並列で動かせます。

```bash
# オーケストレーターが別エージェントを起動する
tmux send-keys -t multiagent:worker1 \
  "claude --dangerously-skip-permissions -p 'APIエンドポイントの実装'" Enter
```

VS Codeの統合ターミナルでも複数タブは開けます。ただ、Claude Codeを何セッションも並列で動かすと、VS Code自体がメモリを食います。tmuxを使ったほうが軽快です。

リモートサーバー上でClaude Codeを動かす場合も、tmuxがないと作業状態が保持できません。SSH接続が切れると、tmuxのセッションはバックグラウンドで生き続けますが、VSCodeの統合ターミナルは切れます。

## 設計思想の転換

IDE（統合開発環境）は、エンジニアが直接操作する前提で設計されています。補完はキーを打つ人を助けるためにあり、デバッガはコードを読む人を助けるためにあります。

Claude Codeのような**エージェント型**のツールを主軸に置くと、前提が変わります。実装するのは人間ではなくエージェントです。エージェントにはハイライトや補完は不要です。エージェントはファイルを直接読み書きし、コマンドを実行し、結果を確認します。

Copilotも2025年にAgent Modeを追加し、エージェント的な機能を備えつつあります。ただし、その動作基盤はVS Codeの中です。補完もAgent Modeも、IDE統合が前提の設計です。

どちらが良いかという話ではなく、どのAIツールをどう使うかで、自然と合う開発環境も変わってきます。Claude Codeをメインに据えるなら、ターミナル中心のほうが自然に収まります。

「Cursorで動かすのは情弱」——この言い方、好きではありません。選択の問題なので。ただ、エージェント型を主軸にするならIDEとの相性が下がる、という構造的な話は正しいと思います。

## まとめ

- 現在の開発スタイルはWSL2 + tmux + Claude Codeで、VS Codeはほぼ起動しない
- Claude Code自体がファイル操作・コード検索・Git操作を担うため、IDEの出番が減った
- tmuxとの組み合わせで複数エージェントの並列実行が容易になる
- エージェント型AIを主軸にすると、IDE中心の設計思想とずれてくる

このまま使い続けながら、どこまでターミナルで完結できるか様子を見ています。
