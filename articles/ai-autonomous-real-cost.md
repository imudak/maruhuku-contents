## はじめに：理想と現実のギャップ

先日公開した記事「[AIに仕事を任せる技術 — OpenClaw×Claude Codeで実現する自律開発ワークフロー](https://zenn.dev/imudak/articles/ai-autonomous-dev-practice)」では、AI自律運用のワークフローを解説しました。司令塔AIが判断し、複数のサブエージェントが並列で実装を進める。まるで小さなチームのように。

しかし、運用を始めて10日後、現実が突きつけられました。

```
Claude API使用量: 94% → 100%
ステータス: 自律実行停止
```

前記事で語った「理想」を実践した結果、月額$20の枠が10日でパンクしました。そして筆者（imudak / <!-- textlint-disable ja-technical-writing/max-kanji-continuous-len -->合同会社花丸福<!-- textlint-enable ja-technical-writing/max-kanji-continuous-len -->）は、$200/月のプランにアップグレードするという判断を迫られました。

この記事では、AI自律運用のリアルなコストを数字とともに振り返ります。10日間で何を成し遂げ、なぜ$200という投資を決断したのか。モデル使い分けやトークン節約のテクニックも含めて、実践報告としてまとめます。

## コストの壁 — 94%からの急転直下

### 2月14日 20:44 — 警告

imudakが使用量94%を検知しました。月の半ばでこの数字です。数日以内に100%到達は明白でした。

### 2月15日 13:52 — 停止

```
Claude API使用量: 100%
ステータス: Rate limit reached
次回リセット: 2月末
```

自律実行が完全に停止しました。

100パターン自律実行の定時ハートビート（05:45, 06:45, 07:45, 13:45）は、すべて`Rate limit`エラー。sessions_spawnやClaude Codeを含め、全モデルがcooldownに入りました。

### 何が枠を消費したのか

振り返ると、トークン消費の主要因は以下でした。

**1. 100パターン自律実行の定時ハートビート**
- 1日4回 × 10日 = 40回の実行
- 各回でネタ元巡回（5〜13件）+ sessions_spawn起動
- Notion DB読み書きの繰り返し

**2. sessions_spawnの並列実行**
- brain-cloneで3調査を並列実行（競合/マネタイズ/MVP）
- utascript、Pitch Trainer Webの並列MVP開発
- ネタ元巡回サブエージェント（1回あたり5〜10セッション）

**3. Claude Code agent teamsの連続起動**
- mirushogi Remotion（4フェーズ）
- brain-clone（Next.js + Prisma + API）
- flow-manager MVP開発

**4. 本セッションでのOpus使用**
- #合同会社チャンネルでOpus 4.6をデフォルト設定
- 判断・設計フェーズでOpusを積極使用

特にOpusの単価が高い（Sonnetの数倍）ため、本セッションでの長時間Opus使用が大きく響きました。

## $200の判断 — ROIの思考実験

枠が尽きた瞬間、選択肢は2つありました。

1. **$20プランを維持** — 月末までAI自律実行を停止、手動開発に戻る
2. **$200プランにアップグレード** — 自律実行を継続

筆者は即座に2を選びました。

### 「月末まで手動に戻る」を選ばなかった理由

$20プランを維持して月末まで手動開発に戻す選択肢もありました。しかし、それでは自律運用の「継続性」が失われます。

**失われるもの：**
- 定時ハートビートによるネタ巡回のリズム
- sessions_spawnで育てた調査パターンの蓄積
- Claude Code agent teamsの実践ノウハウ
- 100パターン自律実行の勢い

月末まで2週間、AI実行を止めれば、運用の習慣自体が途切れます。リセット後も、「枠を気にして萎縮する運用」に戻るだけです。

自律運用は、継続してこそ価値がある。そう判断しました。

## モデル使い分け戦略 — OpusとSonnetの棲み分け

$200プランでも、コスト意識は必要です。無限にトークンを使えるわけではありません。

### Opusを使うべき場面

**Opus 4.6の強み：**
- 判断・設計フェーズでの深い思考
- 複雑な文脈の理解（100パターン自律実行の全体把握等）
- マルチエージェント協働の司令塔役

**実際の使い分け：**

| シーン | モデル | 理由 |
|--------|--------|------|
| 新規プロジェクト企画 | Opus | 判断・優先順位付けが必要 |
| 100パターン自律実行 | Opus | 全体把握・タスク振り分けが複雑 |
| 設計ドキュメント作成 | Opus | 文脈理解・構造化が重要 |
| コード実装（Claude Code） | Sonnet | ファイルI/O中心、単価が安い |
| sessions_spawn調査 | Sonnet | Web検索・要約中心、並列実行向き |
| 定型タスク（ネタ巡回等） | Sonnet | パターン化済み、量が多い |

### sessions_spawnの並列実行でSonnetを使う

sessions_spawnの真価は並列実行にあります。競合調査・マネタイズ調査・MVP優先順位を3つ同時に走らせられます。

しかし、並列で3つのOpusセッションを起動すれば、トークン消費は3倍です。調査タスクはSonnetで十分な品質が得られるため、sessions_spawnはSonnetをデフォルトにしました。

**例：brain-cloneの並列調査（Sonnet）**
- 3セッション × 約10分 = 合計30分（並列なので実時間10分）
- トークン消費はOpusの約1/3

Opusで逐次実行すれば30分 × 3 = 90分かかり単価も高い。Sonnetで並列実行すれば実時間10分で単価も低い。時間とコストの両方で大きな差がつきます。

### Claude Code agent teamsは常にSonnet

Claude Code agent teamsは、内部で複数エージェントを並列起動します。mirushogiのPhase 1では、setup-agentとtypes-agentが同時に動きました。

agent teams全てがOpusで動くと、トークン消費が跳ね上がります。Claude Codeのデフォルトモデルは設定可能なので、Sonnetを標準にしました。

判断が必要な場面だけ、手動でOpusを指定しています。

## トークン節約テクニック — 実際に効果があったもの

### 1. MEMORY.mdのコンパクション

OpenClawのセッションは、MEMORY.mdを毎回読み込みます。このファイルが肥大化すると、セッション開始のたびに大量のトークンを消費します。

**対策：**
- 週次でMEMORY.mdを整理し、不要情報を削除
- 古い内容はアーカイブして、長期記憶のみ残す
- 日次ログは `memory/YYYY-MM-DD.md` に分離

**効果：**
- MEMORY.mdを5000字から3000字に削減
- セッション開始時のトークン消費を約40%削減

### 2. sessions_spawnのラベル管理

sessions_spawnはラベルで再接続できます。同じタスクを複数回起動すると、過去のセッションが溜まります。

**対策：**
- タスクごとに一意のラベルを付ける（例：`neta-patrol-0745`）
- 完了したセッションは次回起動時に上書き（同じラベルを再利用）
- 並列実行の上限を意識（3〜5セッション程度）

**効果：**
- 不要なセッション履歴の蓄積を防止
- 再接続時のコンテキスト読み込みを最小化

### 3. Claude Codeの確認プロンプト削減

Claude Code agent teamsは、確認プロンプトを挟むとトークンが無駄になります。`--dangerously-skip-permissions`フラグで確認をスキップし、一気に実行させます。

**前提：**
- jj（Jujutsu）環境で、いつでも`op restore`で巻き戻せる
- 禁止操作は最小限（認証情報、OpenClaw設定、システム破壊コマンド）に絞る

**効果：**
- 確認プロンプトのラウンドトリップ削減
- 実行時間短縮 → トークン消費削減

### 4. 設計ドキュメントの事前作成

Claude Code agent teamsに「このプロジェクトを実装して」と大雑把に指示すると、矛盾するコードを同時に生成して迷走します。

**対策：**
- 必ず`docs/<feature>-design.md`で設計を固めてから実装
- ファイル構造、責務分担、API仕様を明記
- agent teamsには「このドキュメントに従って実装せよ」と指示

**効果：**
- 迷走による無駄なトークン消費を防止
- 実装速度が向上（mirushogi 12分の事例）

### 5. ハートビートの頻度調整

100パターン自律実行は、当初1時間ごとに設定していました。しかし、1日24回のハートビートはトークン消費が激しいです。

**対策：**
- ハートビートを1日4回に削減（05:45, 06:45, 07:45, 13:45）
- 各回の実行内容を濃くする（ネタ巡回 + 進行中タスク確認 + 新規採用）

**効果：**
- ハートビート回数を24回/日から4回/日に削減（83%減）
- 1回あたりの処理量は増えるが、起動オーバーヘッドが支配的なため、全体では約70%のトークン削減

## 学んだこと — AI自律運用のコスト構造

### コストは「開発速度」に比例する

AI自律運用のコストは、「どれだけ速く開発するか」に比例します。

- 手動開発：コスト$0、速度1x
- AI補助開発：コスト$20/月、速度3x
- AI自律運用：コスト$200/月、速度10x

速度を求めるほどコストは上がります。ただ、「時間をお金で買う」という視点では、個人開発者が外注するよりはるかに安い投資です。

### 個人開発者/一人法人のAIコスト判断

筆者がアップグレードを決めた条件は以下の通りです。

1. **時間が最も貴重なリソース** — 一人法人で全てを自分でやる必要がある
2. **アイデアの探索量を増やしたい** — 100パターンのマネタイズを試したい
3. **投資を回収する見込みがある** — gachi-todoで広告収入の実績がある
4. **法人として経費計上できる** — 合同会社の経費として処理可能

逆に、趣味開発やお試し段階では$20プランで十分だと思います。コストが気になって萎縮するくらいなら、まず$20プランで「自律運用とはどういうものか」を掴んでからアップグレードを検討する方が健全です。
