## はじめに：理想と現実のギャップ

先日公開した記事「[AIに仕事を任せる技術 — OpenClaw×Claude Codeで実現する自律開発ワークフロー](https://zenn.dev/maruhuku/articles/ai-autonomous-dev-practice)」では、**AI自律運用の理想的なワークフロー**を解説しました。司令塔AIが判断し、複数のサブエージェントが並列で実装を進める。まるで小さなチームのように。

しかし、運用を始めて10日後、**現実が突きつけられました。**

```
Claude API使用量: 94% → 100%
ステータス: 自律実行停止
```

**前記事で語った「理想」を実践した結果、月額$20の枠が10日でパンクしました。** そして筆者（imudak / 合同会社花丸福）は、$200/月のプランにアップグレードするという判断を迫られました。

この記事では、AI自律運用の**リアルなコスト**を数字とともに振り返ります。10日間で何を成し遂げ、なぜ$200という投資を決断したのか。モデル使い分けやトークン節約のテクニックも含めて、**等身大の実践報告**をお届けします。

## コストの壁 — 94%からの急転直下

### 2月14日 20:44 — 警告

imudakが使用量94%を検知しました。この時点で、すでに**月額枠の大半を消費**していました。

月の半ばで94%。このペースだと、**数日以内に100%に到達する**ことは明白でした。

### 2月15日 13:52 — 停止

```
Claude API使用量: 100%
ステータス: Rate limit reached
次回リセット: 2月末
```

**自律実行が完全に停止しました。**

100パターン自律実行の定時ハートビート（05:45, 06:45, 07:45, 13:45）は、すべて`Rate limit`エラー。sessions_spawnもClaude Codeも、Opus も Sonnetも、全モデルがcooldownに入りました。

### 何が枠を消費したのか

振り返ると、トークン消費の主要因は以下でした。

**1. 100パターン自律実行の定時ハートビート**
- 1日4回 × 10日 = 40回の実行
- 各回でネタ元巡回（5-13件）+ sessions_spawn起動
- Notion DB読み書きの繰り返し

**2. sessions_spawnの並列実行**
- brain-cloneで3調査を並列実行（競合/マネタイズ/MVP）
- utascript、Pitch Trainer Webの並列MVP開発
- ネタ元巡回サブエージェント（1回あたり5-10セッション）

**3. Claude Code agent teamsの連続起動**
- mirushogi Remotion（4フェーズ）
- brain-clone（Next.js + Prisma + API）
- flow-manager MVP開発

**4. 本セッションでのOpus使用**
- #合同会社チャンネルでOpus 4.6をデフォルトに設定
- 判断・設計フェーズでOpusを積極使用

特に、**Opusの単価が高い**（SonnetやHaikuの数倍）ため、本セッションでの長時間Opus使用が大きく響きました。

## $200の判断 — ROIの思考実験

枠が尽きた瞬間、選択肢は2つありました。

1. **$20プランを維持** — 月末までAI自律実行を停止、手動開発に戻る
2. **$200プランにアップグレード** — 自律実行を継続

筆者は即座に**2. アップグレード**を選びました。なぜか。

### 一人法人としての投資判断

合同会社花丸福は一人法人です。開発リソースは「imudak 1名」のみ。**AI自律運用は、実質的に「開発者を雇う」ことに等しい投資**です。

**投資対効果の試算：**

| 項目 | 手動開発 | AI自律運用 |
|------|---------|-----------|
| brain-clone開発時間 | 2-3日 | 1日（1.5時間実装） |
| mirushogi Remotion | 数時間 | 12分 |
| 100パターンネタ巡回 | 手動調査1件30分 | 自動巡回1日13件 |
| 月間コスト | $0 | $200 |

**単純計算：**
- 手動開発: 1プロジェクト = 3日 → 月10営業日 = 約3プロジェクト
- AI自律運用: 10日で38プロジェクト → 月換算で約114プロジェクト

もちろん、全てがMVP完成するわけではありません。百式登録（企画段階）が大半です。しかし、**アイデアの探索速度が10倍以上**になっているのは事実です。

**ROIで考えれば：**
- $200/月 ≒ 時給$2500の開発者を1日だけ雇う（8時間換算）
- しかしAI自律運用は24時間稼働可能
- 実質的な時給は**$0.28/時**（$200 ÷ 720時間）

### 「枠が尽きたら手動に戻る」という選択肢を捨てた理由

$20プランを維持して、月末までAI停止という選択もありました。しかし、**それでは自律運用の「継続性」が失われます。**

**失われるもの：**
- 定時ハートビートによるネタ巡回の継続性
- sessions_spawnで育てた調査パターンの蓄積
- Claude Code agent teamsの実践ノウハウ
- 100パターン自律実行のリズム

月末まで2週間、AI実行を止めれば、**運用の習慣自体が途切れます。** そしてリセット後も、また「枠を気にして萎縮する運用」に戻るだけ。

**自律運用は、継続してこそ価値がある。** そう判断しました。

## モデル使い分け戦略 — OpusとSonnetの棲み分け

$200プランにアップグレードしても、**コスト意識は必要**です。無限にトークンを使えるわけではありません。

### Opusを使うべき場面

**Opus 4.6の強み：**
- 判断・設計フェーズでの深い思考
- 複雑な文脈の理解（100パターン自律実行の全体把握等）
- マルチエージェント協働の司令塔役

**実際の使い分け：**

| シーン | モデル | 理由 |
|--------|--------|------|
| 新規プロジェクト企画 | Opus | 判断・優先順位付けが必要 |
| 100パターン自律実行 | Opus | 全体把握・タスク振り分けが複雑 |
| 設計ドキュメント作成 | Opus | 文脈理解・構造化が重要 |
| コード実装（Claude Code） | Sonnet | ファイルI/O中心、単価安い |
| sessions_spawn調査 | Sonnet | Web検索・要約中心、並列実行 |
| 定型タスク（ネタ巡回等） | Sonnet | パターン化済み、量が多い |

### sessions_spawnの並列実行でSonnetを使う

sessions_spawnは、**並列実行が真価**です。競合調査・マネタイズ調査・MVP優先順位を3つ同時に走らせる。

しかし、並列で3つのOpusセッションを起動すれば、**トークン消費は3倍**になります。調査タスクは、Sonnetで十分な品質が得られるため、**sessions_spawnはSonnetをデフォルト**にしました。

**例：brain-cloneの並列調査（Sonnet）**
- 3セッション × 約10分 = 合計30分（並列なので実時間10分）
- トークン消費: Opusの約1/3

Opusで逐次実行すれば30分 × 3 = 90分 + 高単価。Sonnetで並列実行すれば10分 + 低単価。**時間もコストも3倍以上の差**です。

### Claude Code agent teamsは常にSonnet

Claude Code agent teamsは、**内部で複数エージェントを並列起動**します。mirushogiのPhase 1では、setup-agentとtypes-agentが同時に動きました。

agent teams全てがOpusで動けば、**トークン消費が爆発**します。幸い、Claude Codeのデフォルトモデルは設定可能なので、**Sonnetを標準**にしました。

**判断が必要な場面だけ、手動でOpusを指定**します。

## トークン節約テクニック — 実際に効果があったもの

### 1. MEMORY.mdのコンパクション

OpenClawのセッションは、`MEMORY.md` を毎回読み込みます。このファイルが肥大化すると、**セッション開始のたびに大量トークンを消費**します。

**対策：**
- 週次で `MEMORY.md` を整理し、不要情報を削除
- 「今週の重要事項」セクションを設け、古い内容はアーカイブ
- 日次ログは `memory/YYYY-MM-DD.md` に分離し、MEMORY.mdは「長期記憶」のみ

**効果：**
- MEMORY.md を 5000字 → 3000字に削減
- セッション開始時のトークン消費 約40%削減

### 2. sessions_spawnのラベル管理

sessions_spawnは、**ラベルで再接続**できます。同じタスクを複数回起動すると、過去のセッションが溜まります。

**対策：**
- タスクごとに一意のラベルを付ける（例：`neta-patrol-0745`）
- 完了したセッションは、次回起動時に上書き（同じラベルを再利用）
- 並列実行の上限を意識（3-5セッション程度）

**効果：**
- 不要なセッション履歴の蓄積を防止
- 再接続時のコンテキスト読み込みを最小化

### 3. Claude Codeのチェックポイント削減

Claude Code agent teamsは、**確認プロンプトを挟むとトークンが無駄**になります。`--dangerously-skip-permissions` フラグで確認をスキップし、一気に実行させます。

**前提：**
- jj（Jujutsu）環境で、いつでも `op restore` で巻き戻せる
- 禁止操作は最小限（認証情報、OpenClaw設定、システム破壊コマンド）に絞る

**効果：**
- 確認プロンプトのラウンドトリップ削減
- 実行時間短縮 → トークン消費削減

### 4. 設計ドキュメントの事前作成

Claude Code agent teamsに「このプロジェクトを実装して」と大雑把に指示すると、**矛盾するコードを同時に生成**して迷走します。

**対策：**
- 必ず `docs/<feature>-design.md` で設計を固めてから実装
- ファイル構造、責務分担、API仕様を明記
- agent teamsには「このドキュメントに従って実装せよ」と指示

**効果：**
- 迷走による無駄なトークン消費を防止
- 実装速度が劇的に向上（mirushogi 12分の事例）

### 5. ハートビートの頻度調整

100パターン自律実行は、当初「1時間ごと」に設定していました。しかし、1日24回のハートビートは**トークン消費が激しい**。

**対策：**
- ハートビートを「1日4回」に削減（05:45, 06:45, 07:45, 13:45）
- 各回の実行内容を濃くする（ネタ巡回 + 進行中タスク確認 + 新規採用）

**効果：**
- ハートビート回数 24回/日 → 4回/日（83%削減）
- トークン消費 約70%削減（1回あたりの処理量は増えるが、起動オーバーヘッドが支配的）

## 学んだこと — AI自律運用のコスト構造

### コストは「開発速度」に比例する

AI自律運用のコストは、**「どれだけ速く開発するか」に比例**します。

- 手動開発: コスト$0、速度1x
- AI補助開発: コスト$20/月、速度3x
- AI自律運用: コスト$200/月、速度10x

速度を求めるほど、コストは上がります。しかし、**時間をお金で買う**という視点では、極めて安い投資です。

### 個人開発者/一人法人のAIコスト投資の考え方

**AI自律運用は、以下のような人に向いています：**

1. **時間が最も貴重なリソース**である人
   - 副業で開発時間が限られている
   - 一人法人で全てを自分でやる必要がある
2. **アイデアの探索量を増やしたい**人
   - 100パターンのマネタイズを試したい
   - MVP量産でヒットを狙う
3. **初期投資を回収する見込みがある**人
   - 既にマネタイズ実績がある（広告収入、アプリ売上等）
   - 法人として経費計上できる

**逆に、以下のような人には勧めません：**

- AIを「お試し」で使いたいだけ
- マネタイズの見込みが全くない趣味開発
- $200/月の投資を「高い」と感じる

**筆者の場合：**
- 合同会社として経費計上可能
- gachi-todoで広告収入の実績あり
- 100パターン探索で次のヒットを狙う戦略

→ $200/月の投資は「経営判断として妥当」と判断しました。

## まとめ — コストと向き合う

AI自律運用のリアルは、**美しい理想だけでは語れません。**

- 10日で38プロジェクトを爆誕させ
- brain-cloneを1日で、mirushogiを12分で実装し
- 100パターン自律実行で毎日13件のアイデアを巡回する

この「理想」の裏側には、**$20 → $200という10倍のコスト増**がありました。

しかし、筆者はこの投資を後悔していません。なぜなら、**時間は買えるが、機会は二度と来ない**からです。

AI自律運用は、個人開発者が「小さなチーム」のように働ける唯一の手段です。$200/月で10倍の開発速度を手に入れられるなら、それは**破格の投資**です。

**ただし、コストと向き合う覚悟は必要です。**

- トークン節約のテクニックを駆使し
- OpusとSonnetを使い分け
- 無駄なセッションを削減し
- 設計ドキュメントで迷走を防ぐ

この記事が、AI自律運用を検討している個人開発者・一人法人の方々にとって、**等身大の判断材料**になれば幸いです。
