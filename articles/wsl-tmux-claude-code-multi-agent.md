tmuxで9つのペインが一気に展開されて、それぞれにClaude Codeが起動し始めました。「将軍」「家老」「足軽1〜8」の役割名が並ぶ。これが[multi-agent-shogun](https://github.com/yohey-w/multi-agent-shogun)の起動シーンです。

ちょっと笑えるビジュアルですが——実際にタスクを渡すと、9体のAIが黙々と動き始めました。

## きっかけ

[@shio_shoppaize](https://zenn.dev/shio_shoppaize)さんの記事「Claude Codeで『AI部下10人』を作ったら、勝手にバグ直して『違反は切腹』ルールを追加してきて、オレは適当にしゃべるだけになった」を読みました。

タイトルだけ見ると煽り系ですが、内容は実装の話です。tmux上で複数のClaude Codeプロセスを起動して、YAMLファイルでタスクをやり取りさせる。将軍がタスクを受け取り、家老が分解して、8人の足軽が並列実行する。そういう構成のOSSを作って実際に動かした話でした。

WSL2とtmuxは日常的に使っています。Claude Codeも使っています。「forkして試すだけなら大した手間じゃない」と判断して、やってみることにしました。

## 環境とセットアップ

環境はWindows 11 + WSL2 + Ubuntu。tmuxは長年使っているので別途インストールは不要でした。

```bash
git clone https://github.com/yohey-w/multi-agent-shogun
cd multi-agent-shogun
```

設定ファイルを自分の環境に合わせて修正して、起動スクリプトを実行します。

```bash
./shutsujin_departure.sh
```

これだけで、tmuxセッションが作成されて9つのペインにClaude Codeが起動します。各ペインには担当エージェントの指示書（CLAUDE.md）が読み込まれていて、役割がすでに設定されています。セットアップ自体は1時間もかかりませんでした。

## 設計を読む

タスクを渡してから最初に気づいたのは、エージェント間の通信がファイルベースだということです。

```text
queue/
  tasks/
    karo.yaml        ← 将軍→家老へのタスク
    ashigaru_1.yaml  ← 家老→足軽1へのタスク
    ...
```

将軍がYAMLに書いて、tmux send-keysで家老を起こす。家老がYAMLを読んで処理して、足軽に割り振る。足軽が完了したらステータスをYAMLに書く。

状態共有にはdashboard.mdを使います。

```markdown
## 🔄 進行中
| 足軽 | タスク | 進捗 |
|------|--------|------|
| 足軽1 | API設計 | 実行中 |
| 足軽3 | テスト作成 | 実行中 |

## ✅ 本日の戦果
| 時刻 | 任務 | 結果 |
|------|------|------|
| 14:15 | DB設計 | ✅ 完了 |
```

dashboard.mdの更新者は家老だけと決まっています。単一ライターにすることで競合を防ぐ設計です。

## ハマったポイント

動かし始めてすぐに問題が出ました。tmux send-keysの挙動です。

```bash
# ❌ これが動かない
tmux send-keys -t multiagent:0.0 'タスクを実行せよ' Enter

# ✅ 2回に分ける
tmux send-keys -t multiagent:0.0 'タスクを実行せよ'
tmux send-keys -t multiagent:0.0 Enter
```

一行で書くと`Enter`がテキストとして送られてしまい、Claude Codeが反応しません。shogunの指示書にはこれが「絶対禁止パターン」として明記されていました。後から読めば書いてあったのですが、先に動かして確認してしまいました。

もう1つは足軽間のファイル競合です。8体が並列で動くと、同じファイルに書こうとする場面が出てきます。shogunではエージェントごとに専用ファイルを使う設計（output_1.md、output_2.md...）で対処していて、競合の可能性がある場合は即座に`blocked`報告を上げる仕組みも入っています。

## 「自律的に動く」の中身

将軍にバグレポートを渡したときの流れはこうなります。

1. 将軍がバグの内容を確認してYAMLに記録する
2. 家老が調査・修正・テストに分解して足軽に割り振る
3. 足軽が並列で担当箇所を処理する
4. 家老が完了報告をdashboard.mdに記録する
5. 将軍が結果を確認する

人間がやることは「バグレポートを渡す」と「完了報告を確認する」だけです。各ステップで人間が次の指示を出す必要がない。エージェントが受け取った役割に従って次に何をするかを自分で判断します。

これが「自律的に動く」の実態です。

## AIがルールを追加した、という話

オリジナルの記事に「足軽たちが自分たちで『他の担当領域を侵害しない』というルールを書き加えた」というエピソードが出てきます。いわゆる切腹ルールです。

forkしたリポジトリのCLAUDE.mdには、そのルールがすでに書かれています。記事のAIが自律的に追加したルールが、そのままコードに残っている。

自分が動かしたエージェントたちはそのルールに従って動いていました。ある足軽が「このタスクは足軽3の担当範囲と重複する可能性がある」として`blocked`報告を上げる場面がありました。ルールを書いたのはオリジナル作者のAIですが、それが実際に機能している様子を見ると、「AIが自律的にガバナンスを整備した」という話の意味が少し実感できます。

## イベント駆動の重要性

ポーリング方式で実装すると、9体のエージェントが定期的にAPIを叩き続けることになります。それだけでコストが跳ね上がります。

shogunがtmux send-keysを使うのはこのためです。エージェントは基本的に停止状態にあって、起こされたときだけ動く。コストはタスクを処理している時間分だけです。

Claude Codeは対話型AIなので、プロンプトが表示されている間は「待機中」ではなく「停止中」です。ループさせるとAPI代が飛ぶ。だからイベント駆動にするしかない。tmux send-keysというのは地味な仕組みですが、コスト面では核心的な設計判断です。

## 2週間運用して

実際に2週間ほど運用しました。

感触としては「動く。ただし指示の質が直接アウトプットに反映される」でした。将軍への指示が曖昧だと家老の分解が的外れになり、足軽がずれた方向に進みます。コスト面はClaude Max($100〜200/月)で10エージェント運用できる試算があり、イベント駆動のおかげで想定外の超過にはなりませんでした。

その後、Claude Code本体にオーケストレーション機能が追加されました。将軍・家老の役割がプラットフォーム側に吸収され、shogunが解決していた問題の多くが自然に消えました。導入から2週間後に移行しましたが、実際に動かした経験は移行先の設計判断に役立っています。詳細は[別の記事](https://zenn.dev/imudak/articles/multi-agent-orchestration-comparison)に書きました。

## まとめ

WSL2 + tmux + multi-agent-shogunの組み合わせは、追加ツールなしで10エージェントのマルチエージェント環境を作れます。

実際に動かして確認できたことは次の通りです。

- エージェント間通信はファイルベースが安定する
- イベント駆動（tmux send-keys）でコストを抑える
- 単一ライターで競合を防ぐ（dashboard.mdは家老だけが更新）
- 担当外への干渉を自律的に判断させることが実用化の鍵

今から始めるなら、Claude Codeの組み込みオーケストレーション機能のほうが手軽です。ただ、tmuxベースの仕組みを一度動かしてみると、マルチエージェントの基礎的な問題（競合、通信、待機コスト）が体感できます。高レベルのツールを使う前に仕組みを理解しておくと、トラブルシューティングの速さが変わると感じました。
