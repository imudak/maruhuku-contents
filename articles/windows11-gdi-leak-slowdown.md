CPU 4%、メモリ半分、ディスクも余裕あり。なのにマウスがカクつく。仮想デスクトップの切り替えが遅い。

4日ほど再起動せずに使い続けていたら、こういう状態になりました。タスクマネージャーを見ても何もおかしくないのに、UIだけが重い。調べていくと、見慣れない指標が原因でした。

## 環境

- Windows 11 Pro
- Intel Core Ultra 7 265（20コア）/ RAM 64GB
- 常駐: Chrome、VMware、VS Code、Teams、Discord
- システム稼働時間: 約4日

## タスクマネージャーでは何もわからなかった

最初にタスクマネージャーを開きました。

| 項目 | 値 |
|------|-----|
| CPU | 4% |
| メモリ | 32.7GB / 63.5GB（51%） |
| ディスク（C:） | 空き461GB |
| GPU | 低い |

どこにもボトルネックがありません。「プロセス」タブのデフォルト列はCPU・メモリ・ディスク・ネットワークの4つで、UI描画の負荷に関する情報は出てきません。

## GDIオブジェクトという存在を知る

調べていくうちに「GDIオブジェクト」という指標にたどり着きました。

GDI（Graphics Device Interface）は、WindowsのUI描画の基盤APIです。ウィンドウの枠、文字、アイコン、ボタン——画面に表示されるほぼ全ての要素はGDIを通じて描画されます。描画に使われるリソース（フォント、ブラシ、ペン、ビットマップ等）が「GDIオブジェクト」として管理されています。

ポイントは2つあります。

- **プロセスあたりの上限が10,000。** アプリが描画用リソースを確保した後、不要になっても解放しないと蓄積していく
- **描画処理はOS全体で共有するカーネルリソースを経由する。** 一部のプロセスの蓄積がシステム全体のUI応答性に影響する

つまり、CPU・メモリには余裕があっても、GDIオブジェクトが溜まるとUIだけが重くなるわけです。

## GDI列をタスクマネージャーに追加する

タスクマネージャーでGDIオブジェクト数を確認できます。デフォルトでは非表示なので、列を追加します。

1. タスクマネージャーを開く
2. **「詳細」タブ**を選択
3. 列ヘッダーを右クリック →「列の選択」
4. **「GDIオブジェクト」**にチェックを入れる

## 犯人が見えた

GDI列を追加して降順ソートすると、一目瞭然でした。

| プロセス | GDIオブジェクト数 | 親アプリ |
|----------|--:|----------|
| msedgewebview2 | **6,856** | Microsoft Teams |
| chrome | **6,704** | Google Chrome |
| msedgewebview2 | **6,694** | （別アプリ） |
| msedgewebview2 | **6,691** | （別アプリ） |
| vmware | 1,755 | VMware（正常範囲） |

上位4プロセスがいずれも上限10,000の67〜69%に達していました。

### msedgewebview2の親を辿る

`msedgewebview2.exe`はアプリ本体ではありません。**Edge WebView2ランタイム**を埋め込んでいるアプリのレンダリングプロセスです。タスクマネージャーの「詳細」タブだけではどのアプリが原因なのかわかりません。

PowerShellで親プロセスを辿りました。

```powershell
Get-CimInstance Win32_Process -Filter "ProcessId = 23804" |
  Select-Object Name, ProcessId, ParentProcessId
```

この結果、GDI 6,856を消費していた`msedgewebview2`の親はMicrosoft Teamsでした。CopilotもWebView2を使っており、同様にGDIを蓄積していました。

## 段階的に終了して改善を確認

以下の順にアプリを終了し、そのつど体感を確認しました。

| 手順 | 操作 | 効果 |
|:--:|------|------|
| 1 | Explorer.exeを再起動 | 限定的 |
| 2 | Microsoft Copilot（365版 / 通常版）を終了 | **体感で明確に改善** |
| 3 | Google Chromeを完全終了 | GDI 6,704解消 |
| 4 | Microsoft Teamsを終了 | GDI 6,856解消 |

対処後のGDI上位プロセスは以下の通りです。

| プロセス | GDIオブジェクト数 |
|----------|--:|
| vmware | 1,755 |
| eguiProxy（ESET） | 308 |
| explorer | 216 |
| tailscale-ipn | 125 |

全プロセスが正常範囲内に収まり、UI操作の遅延は完全に解消されました。

## GDI以外の蓄積

GDIリークのほかにも、長時間稼働による蓄積がありました。

| 項目 | 値 | 備考 |
|------|-----|------|
| Nonpaged Pool | 1,733 MB | やや高め |
| Paged Pool | 2,252 MB | 蓄積傾向あり |
| Explorer.exeハンドル数 | 6,809 | 通常より多い |

これらはアプリの再起動では解消されず、OS再起動が必要です。

## 再発防止

今回の対処で学んだことをまとめます。

### アプリの定期的な再起動

Teams・Chrome・Copilotは長時間起動でGDIリークが蓄積します。毎日の業務終了時に閉じる習慣をつけるだけで違います。

### 「再起動」と「シャットダウン」の違いに注意

Windows 11はデフォルトで**高速スタートアップ**が有効です。この状態では「シャットダウン」がカーネルセッションを保存して復元するため、Nonpaged/Paged Poolが完全にはリセットされません。

完全にリセットするには「シャットダウン」ではなく**「再起動」**を使います。

### Edge WebView2ランタイムの更新

GDIリークはWebView2ランタイム側のバグの可能性もあります。`winget`で定期的に更新しておくのがよさそうです。

```powershell
winget upgrade --all
```

### GDI監視の目安

タスクマネージャーの「詳細」タブにGDI列を追加しておけば、いつでも確認できます。**プロセスあたり5,000を超えたら**アプリの再起動を検討するのがよいと思います。

## まとめ

「スペックは足りているのに重い」とき、CPU・メモリ・ディスクだけ見ていても原因にたどり着けないことがあります。今回の流れをまとめると以下の通りです。

1. タスクマネージャーの「詳細」タブでGDI列を追加
2. GDIオブジェクト数が異常に多いプロセスを特定
3. `msedgewebview2`は親プロセスを辿って真犯人を特定
4. 該当アプリを再起動して解消

Teams・Chrome・Copilotのように、WebView2ベースのアプリを常時起動している環境では起こりやすい問題のようです。「最近なんか重いな」と思ったら、GDI列を見てみてください。
