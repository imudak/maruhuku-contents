CPU 4%、メモリ半分、ディスクも余裕。なのにマウスがカクつく。仮想デスクトップの切り替えが遅い。

「スペックは足りてるのに重い」という状況に遭遇したら、GDIオブジェクトのリークを疑ってみてほしい。

## 環境と症状

- Windows 11 Pro
- Intel Core Ultra 7 265（20コア）/ RAM 64GB
- システム稼働時間：約4日

症状はシンプルだった。

- マウス操作が引っかかる
- 仮想デスクトップの切り替えがもたつく
- Chrome のタブ数や VMware の使い方は普段と変わっていない

## タスクマネージャーでは原因がわからない

最初にやることは誰でも同じだ。タスクマネージャーを開く。

| 項目 | 値 | 判定 |
|------|-----|------|
| CPU | 4% | 問題なし |
| メモリ | 32.7GB / 63.5GB (51%) | 問題なし |
| ディスク (C:) | 空き 461GB | 問題なし |
| GPU | 低い | 問題なし |

何も問題がない。ここで詰まる人は多いと思う。

タスクマネージャーの「プロセス」タブはCPU・メモリ・ディスク・ネットワークの4列がデフォルトで、UIの描画負荷に関する情報は表示されない。問題は別のところにあった。

## GDIオブジェクトという盲点

GDI（Graphics Device Interface）は、WindowsがUI描画に使う基盤APIだ。ウィンドウの枠、文字、アイコン、ボタン——画面に表示されるほぼ全ての要素はGDIを通じて描画される。

描画の際に使われるリソース（フォント、ブラシ、ペン、ビットマップ等）は「GDIオブジェクト」として管理される。

**プロセスあたりの上限は10,000。** アプリが描画用リソースを確保した後、不要になっても解放しないと「リーク」する。上限に近づくと新しいUI要素の描画が遅延・失敗し始める。

重要なのは、**GDIはプロセス単位で管理されるが、描画処理はOS全体で共有するカーネルリソースを経由する**ということだ。一部のプロセスのリークがシステム全体のUI応答性に影響する。

## GDIオブジェクト数の確認方法

タスクマネージャーで常時確認できる。デフォルトでは非表示なので、列を追加する必要がある。

1. タスクマネージャーを開く
2. **「詳細」タブ** を選択
3. 列ヘッダーを右クリック →「列の選択」
4. **「GDI オブジェクト」** にチェックを入れる

これだけで、各プロセスのGDIオブジェクト数が見えるようになる。

## 犯人を特定する

GDI列を追加して降順ソートすると、状況が一目瞭然だった。

| プロセス | GDIオブジェクト数 | 親アプリ |
|----------|:--:|----------|
| msedgewebview2 | **6,856** | Microsoft Teams |
| chrome | **6,704** | Google Chrome |
| msedgewebview2 | **6,694** | 別アプリ |
| msedgewebview2 | **6,691** | 別アプリ |
| vmware | 1,755 | VMware（正常範囲） |

上位4プロセスがいずれもGDI上限10,000の**67〜69%**に達していた。

### msedgewebview2 の正体

`msedgewebview2.exe` はアプリ本体ではない。**Edge WebView2ランタイム**を埋め込んでいるアプリのレンダリングプロセスとして生成される。つまり、犯人はWebView2を使っているアプリ側だ。

親プロセスの特定には、タスクマネージャーの「詳細」タブでPIDを確認し、PowerShellで親子関係を辿る方法が確実だ。

```powershell
Get-CimInstance Win32_Process -Filter "ProcessId = 23804" |
  Select-Object Name, ProcessId, ParentProcessId
```

この結果、GDI 6,856を消費していた `msedgewebview2` の親が **Microsoft Teams** であることが判明した。

## 対処と結果

以下の順に終了し、段階的に改善を確認した。

### 1. Explorer.exe の再起動

タスクマネージャーからエクスプローラーを再起動。ハンドル・スレッドはリセットされたが、体感の改善は限定的だった。

### 2. Microsoft Copilot の終了

Copilot（365版 / 通常版の2つ）を終了 → **体感で明確に改善**。msedgewebview2 の一部が解消された。

### 3. Google Chrome の完全終了

Chrome を終了 → GDI 6,704 が解消。

### 4. Microsoft Teams の終了

Teams を終了 → GDI 6,856（最大の消費源）が解消。

### 対処後

| プロセス | GDIオブジェクト数 |
|----------|:--:|
| vmware | 1,755 |
| eguiProxy (ESET) | 308 |
| explorer | 216 |
| tailscale-ipn | 125 |

全プロセスが正常範囲内に収まり、**UI操作の遅延は完全に解消**された。

## その他の悪化要因

GDIリークの他にも、長時間稼働による蓄積が見られた。

| 項目 | 値 | 備考 |
|------|-----|------|
| Nonpaged Pool | 1,733 MB | やや高め。ドライバレベルのリソース蓄積 |
| Paged Pool | 2,252 MB | 蓄積傾向あり |
| Explorer.exe ハンドル数 | 6,809 | 通常より多い |

これらはOS再起動でしかリセットされない。

## 再発防止策

### 定期的なアプリ再起動

Teams・Chrome・Copilotは長時間起動でGDIリークが蓄積する。毎日の業務終了時に一度閉じる習慣をつけるだけでも効果がある。

### 「再起動」と「シャットダウン」は違う

Windows 11はデフォルトで**高速スタートアップ**が有効になっている。この状態では「シャットダウン」はカーネルセッションを保存して復元するため、**Nonpaged/Paged Poolが完全にはリセットされない**。

完全にリセットしたい場合は、シャットダウンではなく**「再起動」**を使うこと。

### Edge WebView2 ランタイムの更新

WebView2のGDIリークはランタイム側のバグである可能性がある。`winget` で定期的に更新しておく。

```powershell
winget upgrade --all
```

### GDI監視の習慣化

タスクマネージャーの「詳細」タブにGDI列を追加しておけば、いつでも確認できる。**プロセスあたり5,000を超えたら注意**。アプリの再起動を検討しよう。

## まとめ

「スペックは足りてるのに重い」とき、CPU・メモリ・ディスクだけ見ていても原因にたどり着けないことがある。今回のケースでは、以下の手順で原因特定から解決まで進めた。

1. タスクマネージャーの「詳細」タブでGDI列を追加
2. GDIオブジェクト数が異常に多いプロセスを特定
3. `msedgewebview2` は親プロセスを辿って真犯人を特定
4. 該当アプリを再起動して解消

GDIオブジェクトのリークは、個人開発者やエンジニアに限らず、TeamsやChromeを常時起動しているオフィスワーカーなら誰にでも起こり得る問題だ。「最近なんかWindowsが重い」と感じたら、まずGDI列をチェックしてみてほしい。
